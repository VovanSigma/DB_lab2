# DB_lab2
## 1. Види NoSQL баз

Під час виконання роботи я коротко ознайомився з основними типами NoSQL баз даних:

1. **Key-value сховища**  
   Найпростіший тип NoSQL. Дані зберігаються у вигляді пари ключ–значення.  
   Приклади: Redis, Memcached.  
   Зручно використовувати для кешування, зберігання сесій, лічильників.

2. **Документні бази**  
   Дані зберігаються у вигляді документів (частіше за все JSON або BSON).  
   Документи можуть мати різну структуру, вкладені об’єкти та масиви.  
   Приклади: MongoDB, CouchDB.  
   Підходять для зберігання гнучких напівструктурованих даних (профілі користувачів, замовлення тощо).

3. **Колонкові (column-family) бази**  
   Дані організовані по колонках, а не по рядках. Добре працюють на великих обсягах даних і для аналітичних запитів.  
   Приклади: Apache Cassandra, HBase.

4. **Графові бази**  
   Використовуються для зберігання даних у вигляді вершин і ребер (граф).  
   Зручні там, де важливі зв’язки між об’єктами: соціальні мережі, рекомендаційні системи, маршрути.  
   Приклади: Neo4j, ArangoDB.

У цій лабораторній я використовую:
**MongoDB** як документну NoSQL базу;
**Redis** як key-value NoSQL базу іншого типу.

## 2. чи буде ефективніше винести частину даних в іншу базу?

Моя предметна область система управління лікарнею.  
У такій системі є різні типи даних:

довідники (відділення, лікарі, пацієнти);
госпіталізації, прийоми, рахунки, платежі;
медичні записи (історія хвороби, діагнози, призначення, додаткова інформація).

Для цих даних логічно поділити зберігання так:

1. **Що доцільно залишити в SQL (PostgreSQL):**
   госпіталізації;
   рахунки та позиції рахунків;
   платежі;
   зв’язки між пацієнтами, лікарями, відділеннями.

   Тут важлива жорстка цілісність даних, транзакції, зовнішні ключі, тому реляційна БД підходить краще.

2. **Що доцільно винести в NoSQL (MongoDB):**
    медичні записи пацієнта з усіма діагнозами та призначеннями;
    додаткові параметри (алергії, звички, нотатки лікаря, рідкісні поля, які є не у всіх).

   Ці дані мають гнучку структуру, можуть відрізнятися від пацієнта до пацієнта і часто зберігаються як великий “блок історії хвороби”.  
   У документній базі типу MongoDB такий запис зручно зберігати одним документом, без JOIN’ів.
